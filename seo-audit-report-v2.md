## 0. エグゼクティブサマリー（前回からの変化）

前回監査（2日前）で指摘した **CRITICAL/HIGH 級の3つの穴のうち2つは既に塞がれた** と判断しました（ガイドCTA `//deli/` 修正・ガイド孤立解消・店舗登録LP最適化）。さらに M-C2 都道府県/主要都市 description の手書き化、M-C4 `/about/` ページ新設、L-T4 sitemap priority重み付け、L-S3 近隣エリア水平リンクなど、丁寧なフォロー対応が確認できました。

ただし **「ラブホ系LPの canonical問題」は前回からほぼ変化なし**（loveho/index.astro の canonical はいまだ `https://yobuho.com/deli/?tab=loveho`）。また再監査で新たに浮上した課題が以下です。

**新規・再浮上の3つの主要課題:**

1. **ラブホLP独立化が未完**: 前回 H-T1 で指摘した「ラブホ専用パスでの canonical」が `?tab=loveho` のクエリ付きURL指定のまま。サブドメイン loveho.yobuho.com もこのcanonicalを共有 → 「ラブホ 口コミ」KWでGoogleが正規URLを認識できない構造が継続
2. **本番ビルド済み artifact（index.html 等）のフッターに古いサブドメインリンクが残存**: Astroソースは `/deli/` パスベースに移行済みだが、デプロイ済みHTMLにはまだ `https://deli.yobuho.com/` が残っている → 直近デプロイで反映されているかGSCで要確認
3. **ホテル詳細43,580件 vs 口コミ充足率の構造的アンバランス**: 口コミ0件のホテルが大半を占めるとGoogleが「内容が薄い」とみなしインデックス被覆率が下がる懸念。前回M-C5で軽く触れたが、再監査で「店舗登録数増（ビジネス最優先目標）」との連動の観点で再評価

## 2. 新しく見つかった課題

### 新規 N1（HIGH・易）: ラブホLP canonical問題が未解決
**前回からほぼ変わらず、再指摘**

**現状（`astro-src/src/pages/loveho/index.astro` L:9, 13, 35, 38）:**
```
"url": "https://yobuho.com/deli/?tab=loveho",
canonical="https://yobuho.com/deli/?tab=loveho"
ogUrl="https://yobuho.com/deli/?tab=loveho"
```

**何が問題か（初心者向け解説）:**
- `canonical（カノニカル）` とは Google に「このページの正規URLはこれです」と伝えるタグ。重複コンテンツ判定のSEOパワー集約に使う
- `?tab=loveho` はクエリパラメータ付きURL = JavaScriptが必要なSPA挙動依存（ページロード直後に JS でラブホタブを開く動き）
- Googlebot は「正規URL=JSが必要な不安定なページ」と解釈し、最悪「ソフト404（中身が空に見える）」扱いされる
- さらに loveho.yobuho.com サブドメインも同じcanonicalを指している → サブドメインの被リンク資産が「クロスドメイン正規URL」として認識されにくい

**さらに deli/jofu/same/este の各LP の footerLinks にも残存:**
```
{ href: 'https://yobuho.com/deli/?tab=loveho', label: 'ラブホテル' },  // 5箇所
{ href: 'https://yobuho.com/este/?tab=loveho', label: 'ラブホテル' },  // 1箇所
```
全ジャンルLPからラブホへの内部リンクがこのクエリURL経由で、PageRankが分散する構造です。

**ビジネスインパクト:**
- 主要KW「ラブホ 口コミ」「ラブホ 一人入室」で順位が伸びない構造が継続
- CLAUDE-SEO.md でも明記された「ラブホ 一人入室」KWに対応する独立LPが事実上存在しない
- 想定インパクト: 高（前回H-T1からの再指摘）

**推奨対応:**
- 案A（最善）: `/loveho/` 独立パスを .htaccess に追加し、portal-men.html?tab=loveho にリライトしつつ canonical だけ `/loveho/` に独立化
- 案B（最低限）: loveho/index.astro の canonical を `https://yobuho.com/loveho/` に変更し、`/loveho/` をAstroビルド済みLPの公開先として明示

### 新規 N2（HIGH・易）: ラブホLP の CTA も `?tab=loveho` のまま
**新規発見**

loveho/index.astro L:59, 112 の CTA ボタン:

- ジャンル横断回遊としては成立するが、LPコンバージョン経路の純度が下がる

**ビジネスインパクト:**
- 想定インパクト: 中（CTR/直帰率の数字に出る可能性）

**推奨対応:**
- N1 と連動して `/loveho/` 独立パス整備後に CTA も差し替え

### 新規 N3（HIGH・易）: 本番ビルド済 index.html / contact.html などにサブドメインリンク残存
**新規発見**

本番デプロイ済みファイル `index.html`、`contact.html`、`terms.html`、`privacy.html`、`shop-register.html`、`legal.html` のフッターに以下が残存:

```html

<a href="https://jofu.yobuho.com/">女性用風俗専用</a>
<a href="https://same.yobuho.com/">同性利用専用</a>
<a href="https://loveho.yobuho.com/">ラブホテル検索</a>
```

**何が問題か:**
- Astro ソースは `Footer.astro` で `/deli/` `/jofu/` パスベースに変えてあるが、サーバー側のビルド済みファイル（直近 deploy で上書きされてない部分）に古いHTMLが残っている可能性
- もし本番でこのリンクが生きていると、トップページ等から **5つのサブドメインへの内部リンク発信** が継続している → これらサブドメインは旧来のCloudflareサブドメインで、現状LPとしてのみ活躍中
- サブドメインLPの canonical はパス側を指しているので「重複コンテンツ判定」のリスクは低いが、PageRankがクロスドメインに分散する

**確認方法:**
- `curl -s https://yobuho.com/ | grep "deli.yobuho.com"` で本番反映確認
- 既に直近デプロイで上書きされていれば不要（git statusはclean）

**ビジネスインパクト:**
- 想定インパクト: 中（PageRank分散）

**推奨対応:**
- まず本番反映を curl で確認
- 残っていれば次回デプロイで上書きされるが、心配なら手動でAstro再ビルド → デプロイ

### 新規 N4（MEDIUM・中）: ホテル詳細43,580件のインデックス被覆率と「中身の薄さ」リスク
**新規深掘り**

**現状（portal-seo.php L:602-677）:**
- ホテル詳細ページに SSR で hotel カード + 近隣リンク + ガイド誘導が注入される（ソフト404対策）
- ただしユーザー口コミが0件のホテルが大半（43,580件のうち、実投稿があるのは推測で1〜5%）
- title/description は機械生成: `"{ホテル名} - デリヘルを呼べる？口コミ情報"`

**何が問題か:**
- Google は「実コンテンツの量・独自性」を強く見る
- 43,580件分のホテル詳細URLが、住所・施設名・タイプの違いだけで本文が同じ場合「ニアデュプ（near-duplicate）コンテンツ」と判定される
- 結果: GSCの「クロール済み - インデックス未登録」「検出 - インデックス未登録」が大量発生

**ビジネスインパクト:**
- ホテル名 + 地域名のロングテール検索（"○○ホテル池袋 デリヘル"等）で本来狙える流入を取り逃す
- 想定インパクト: 中〜高（長期的・規模感大）

**推奨対応:**
- 案A: 「口コミ件数 ≥ 1 のホテルのみ index、0件は noindex」のロジックを portal-seo.php に追加
  - 効果: Googleのクロール予算（クロールバジェット）を充実ページに集中
  - 副作用: noindex化したホテルもユーザー検索結果には出る（サイト内検索）
- 案B: 口コミ0件のホテルは description に「呼べた / 呼べなかった」の代わりに「未投稿・最初の口コミを投稿する」CTA文言を入れる（コンバージョン誘導 + 一意性確保）
- 案C: hotel-data.json の段階で「口コミ件数」を持ち、sitemap.xml に **口コミがあるホテルだけ追加** する（推奨）

### 新規 N5（MEDIUM・易）: 機械生成descriptionが残る32都道府県の重複リスク
**新規発見**

portal-seo.php L:106-124 で **15都道府県のみ手書き description**。残り32都道府県は機械生成:
```
{pref}で{label}{verb}ホテルを地域から検索。{desc_detail}
```

**何が問題か:**
- M-C2 で「主要15都道府県」の重複を解消したが、残り32都道府県は全モード共通の機械生成パターン
- 例: 「青森県でデリヘルを呼べるホテルを地域から検索。デリヘルをホテルに呼べるか口コミで確認。直通・カードキー・フロント相談など入り方の実績もわかります。」
- 5モード × 32都道府県 = 160URL が、地域名だけ違うほぼ同じdescription
- ニアデュプ判定リスク（致命的ではないが、CTR は低下）

**ビジネスインパクト:**
- 想定インパクト: 低〜中（主要15都道府県以外は検索ボリュームも小さいので、ROI も低い）

**推奨対応:**
- 案A: 残り32都道府県も「県庁所在地名・主要繁華街名」を1〜2語入れる程度の半自動description（30分作業）
- 案B: 都道府県名で機械生成だが、季節性や交通アクセス情報を1文足す
- 優先度低のため、ROIを見極めてから

### 新規 N6（LOW・易）: 近隣エリアセクションが「市区町村ページのみ」表示で、エリアページではスキップ
**新規発見**

portal-seo.php L:548-570 を読むと、近隣エリアリンクは:
```php
if ($areaKey && isset($areaData['area'][$areaKey]['ct']) && is_array($areaData['area'][$areaKey]['ct'])) {
```
**city ページのみ（`$city` が設定されている時のみ）** 表示される実装。

エリアページ（例: /deli/東京都/東京２３区内）には近隣エリアセクションがありません。

**何が問題か:**
- L-S3 の意図は「市区町村→市区町村の水平リンク」だが、エリア→エリアの水平リンクも同じ価値がある
- 例: 「東京２３区内」エリアページから「東京都下」「神奈川県」エリアページへの誘導があると、Googlebotがエリア階層全体をクロールしやすい

**ビジネスインパクト:**
- 想定インパクト: 低（既にエリアナビUIがあるので致命的ではない）

**推奨対応:**
- 必要なら portal-seo.php の elseif ($area) ブロックに同様のロジックを追加（同じ都道府県の他エリアを最大5件）

## 3. 競合動向（新規KWでの観察）

前回 LP に投入された新KW「**失敗しないデリヘル ホテル選び**」「**フロントで断られた**」「**チェックインを断られた**」 は、競合がほぼ取れていない検索意図特化のKW群です。

**観察ポイント:**
- 「失敗しない」「断られた」というネガティブ起点の検索意図は、ユーザーが既に問題意識を持って能動的に検索する状態 → CVR が高い
- 競合の老舗デリヘル系まとめサイトは「○○エリアの呼べるデリヘル一覧」型の情報網羅型コンテンツが主流で、この検索意図特化コンテンツは少ない
- YobuHo の LP コンバージョン特化文言（"フロントで断られた経験ありませんか？"）は **競合に対する差別化として強い** と評価

**推奨アクション:**
- このKW群は早期にGSCのインプレッション数を観察。3〜4週間後に「failed」「断られ」等が含まれる検索クエリの推移を確認すると効果検証が可能
- 競合がコピーしてくる前に、ガイドページにも「フロントで断られた時の対処法」のロングテール記事を1本追加すると堀がさらに広がる

## 補足: 用語解説（前回からの繰り返し、SEO初心者向け）

- **canonical（カノニカル）**: 「このページの正規URLはこれ」とGoogleに教えるmetaタグ。同じ内容が複数URLにある時、SEOパワーを正規URLに集約する
- **soft 404**: 200レスポンスを返しているが内容が空・薄いとGoogleに判断されたページ。インデックスから外される
- **クロール予算（クロールバジェット）**: Googleが1サイトをクロールする回数/時間の上限。大規模サイトでは「重要なページを優先的にクロールしてもらう」設計が必要
- **ニアデュプリケート**: 同じ内容ではないが、地域名や品名だけ違うほぼ同じ内容のページ。Googleが「中身が薄い」と判定する原因
- **E-E-A-T**: Experience / Expertise / Authoritativeness / Trustworthiness。Googleの品質評価軸
